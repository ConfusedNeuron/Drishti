1. Architecture Overview

The system is organized into six layers. Bloomberg and Zerodha are the only external data sources. The risk engine and quant research engine own all numerical truth. The AI copilot is strictly downstream from both and only explains structured results — it never computes independently.

3. Primary Data Flow

BLOOMBERG BLPAPI / CSV Export
  -> bloomberg_cache (parquet/SQLite, keyed by ticker+date)
  -> price_matrix_builder (aligned daily returns for all symbols)
  -> factor_series_builder (commodity, FX, rate, VIX series)

ZERODHA / CSV / SAMPLE
  -> portfolio_importer
  -> portfolio_normalizer (bloomberg_ticker mapping, sector, asset_type)
  -> portfolio_snapshot (stored in local session cache)

RISK PATH:
  portfolio_snapshot + price_matrix
  -> risk_engine (VaR x3, ES, drawdown, volatility, contribution)
  -> backtest_engine (Kupiec, Christoffersen on historical VaR)
  -> stress_engine (scenario shock application)
  -> structured_risk_response

QUANT RESEARCH PATH:
  factor_series + sector_index_returns + portfolio_returns
  -> factor_research_engine (IC, ICIR, t-stat, Granger)
  -> walk_forward_backtester (OOS Sharpe, IC stability)
  -> ml_engine (HMM regime detection, regime-conditioned VaR)
  -> structured_research_response

COPILOT PATH (optional, downstream from both):
  structured_risk_response + structured_research_response
  -> copilot_context_builder (select relevant metrics, sanitize)
  -> Risk MCP tools (calculate_portfolio_risk, run_factor_research, ...)
  -> LLM (explain structured context, no independent calculation)
  -> safe_risk_answer

PRESENTATION:
  structured_risk_response + structured_research_response
  -> dashboard panels (risk overview, drivers, factor research, copilot)
  -> downloadable memo (PDF/HTML risk and research summary)
Design principle The quant research path is independent from the risk path. Both are independent from the AI path. If the LLM provider or MCP server is unavailable, the risk dashboard and research report must still work fully.

4. Bloomberg BLPAPI Integration Architecture

Bloomberg Terminal access is provided through FRTL, IIM Calcutta. The BLPAPI Python library connects to a locally running Bloomberg terminal session. The system is designed with a two-path architecture to handle network/access variability.

Path A — BLPAPI (programmatic)
  1. Bloomberg Terminal is running on the FRTL machine or accessible on the lab network.
  2. BLPAPI Python client connects to localhost:8194 (default Bloomberg server port).
  3. Service //blp/refdata handles historical and reference data requests.
  4. Cache Manager stores all pulled data as parquet files under data/cache/bloomberg/.
  5. Subsequent requests check cache first; only missing date ranges trigger new API calls.
Path B — CSV export fallback
  1. Export data from Bloomberg terminal manually using BDH() Excel formula or Bloomberg export tool.
  2. Save as CSV files following naming convention: {TICKER}_{FIELD}_{START}_{END}.csv.
  3. Place in data/cache/bloomberg/exports/.
  4. Cache Manager detects and ingests CSV files on startup, converting to parquet.
  5. System operates fully offline from this point — identical to BLPAPI path.

5. Zerodha Integration Flow

Prototype flow
  1. User clicks "Connect Zerodha" or chooses sample/CSV.
  2. Backend redirects to Zerodha login page with API key and state token.
  3. Callback receives request_token. Backend exchanges for access token using API secret.
  4. Fetch /portfolio/holdings and /portfolio/positions.
  5. Map each holding to Bloomberg ticker using instrument master + symbol lookup table.
  6. Token kept only in memory for active session. Discarded after use.
Production flow
  1. Consent screen with explicit data access scopes.
  2. CSRF and state token protection on OAuth callback.
  3. Encrypted token storage with short retention and revocation.
  4. Background refresh respects Kite Connect rate limits.
  5. Every data access event logged with timestamp and scope.

6. MCP Architecture

Two MCP servers are used: Zerodha's Kite MCP for broker connectivity in an AI client, and the custom Drishti Risk MCP for project analytics. The copilot uses both to answer questions about portfolio risk, factor signals, and market context.

Kite MCP (Zerodha)

  • Official Zerodha MCP server — exposes broker account and market tools.
  • Used in demo to show live portfolio or market query via AI client.
  • Runs as a separate local process; kept isolated from reliability path.
  • Not required for risk calculations — dashboard works without it.

Drishti Risk MCP (Custom)

  • Exposes all Drishti analytics as structured MCP tools.
  • Calls deterministic risk, backtest, factor, and ML modules.
  • Returns structured JSON — LLM explains, never computes.
  • Runs as a local Python process alongside the backend.
  • Tools: portfolio risk, VaR, ES, stress, backtest, factor IC, regime, news, memo.
AI Client / Claude with MCP
  -> Kite MCP:      get_holdings, get_positions, get_market_quote
  -> Risk MCP:      calculate_portfolio_risk     -> VaR x3, ES, drawdown, contribution
                    run_var_backtest             -> Kupiec/Christoffersen test results
                    run_factor_research          -> IC, t-stat, Granger, walk-forward Sharpe
                    detect_volatility_regime     -> current regime + regime-conditioned VaR
                    run_stress_test              -> scenario loss + contributors
                    analyze_commodity_spillover  -> sensitivity score + lag findings
                    summarize_news_risk          -> risk-tagged headlines
                    generate_risk_memo           -> structured memo
  -> LLM:           convert structured outputs into safe, cited risk explanation

7. Factor Research Pipeline Architecture

The factor research pipeline is designed as an independent module that can run against any set of signals and target return series. It is parameterized, not hard-coded to specific factors.

FactorResearchPipeline(
  factors       = [brent_return, wti_return, gold_return, copper_return,
                   ng_return, usdinr_return, gind10y_change, vix_change],
  targets       = [NSEOILGS_return, NSEIT_return, NSEMETAL_return,
                   NSEBANK_return, portfolio_return, ...NIFTY50_stocks],
  lags          = [1, 2, 3, 5, 10],
  ic_type       = "spearman",
  granger_maxlag = 10,
  walkforward   = { min_train_days: 252, refit_freq: "monthly" }
)

Pipeline steps:
  1. Construct lagged factor matrix (shift factor returns by lag days)
  2. Align factor matrix with target return matrix on common trading dates
  3. Compute IC per factor-lag-target triple: spearman_corr(signal_t, return_t+lag)
  4. Compute ICIR = mean(IC) / std(IC) over rolling 63-day window
  5. Compute IC t-stat = mean(IC) / (std(IC) / sqrt(n_months))
  6. Run Granger causality VAR test for each factor-target pair at all lags
  7. Walk-forward backtest: expanding window, monthly refit, accumulate OOS returns
  8. Report: ranked signal table, top factor-lag pair, OOS Sharpe, regime breakdown

8. ML and Regime Detection Architecture

HMM Regime Detection Pipeline:
  features = [
    rolling_20d_portfolio_vol,        # from risk engine
    rolling_20d_abs_return_mean,      # from risk engine
    invixn_level,                     # from Bloomberg cache
    invixn_5d_change,                 # derived
    brent_5d_return_zscore,           # from Bloomberg cache
    usdinr_5d_change                  # from Bloomberg cache
  ]

  walk_forward_hmm:
    for each month in backtest window:
      -> fit GaussianHMM(n_components=2) on expanding training window
      -> predict regime for next month (OOS)
      -> label states: high_vol = state with higher emission variance
      -> compute regime-conditioned VaR: VaR_cond = historical_var(
           returns[regime_labels == current_state], confidence=0.99)
      -> accumulate: VaR_breach_rate by regime

  Output:
    -> regime_sequence (T labels: 0=low, 1=high)
    -> current_regime (latest label + probability)
    -> regime_conditioned_var (by state)
    -> oos_breach_rate_by_regime (validation evidence)
ML architecture rule HMM is refit at each time step using only past data. Regime labels used for VaR conditioning are always the OOS predictions, never in-sample. This prevents the "regime rewriting" problem common in naive HMM applications.

9. Deployment View

Local course demo
  • Frontend and backend on localhost.
  • Bloomberg data pre-cached via BLPAPI or CSV export before demo day.
  • SQLite/parquet files under data/cache/.
  • Secrets in local .env only.
  • Risk MCP and Kite MCP as local Python processes.
  • LLM use optional — dashboard works without it.
  • Bloomberg data files gitignored — never committed publicly.
Hosted product
  • Frontend and backend deployed separately.
  • Licensed market data subscription (not Bloomberg terminal).
  • Postgres/Supabase with tenant-aware schema.
  • Cloud secret manager.
  • Background workers for data refresh and model refitting.
  • Risk MCP secured behind authenticated tool access.
  • Audit logs and policy checks for all AI outputs.
