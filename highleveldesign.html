1. Architecture Overview

The system is split into four major layers: user interface, application API, analytics services, and external integrations. The key design rule is that the risk engine owns numerical truth; the AI copilot only explains structured results.

Presentation Layer
Web Dashboard
Portfolio, risk, stress, news, commodity panels.
Copilot UI
Natural-language questions and risk memo generation.
CSV/Sample Loader
Reliable fallback for demo.
Report View
Presentation-ready risk summary.
Application API Layer
Auth Controller
Zerodha redirect and session handling.
Portfolio Controller
Import, normalize, store current snapshot.
Risk Controller
Run risk calculations and return structured outputs.
Copilot Controller
Prepare context and call LLM/MCP tools.
Analytics And Data Layer
Risk Engine
VaR, ES, stress, contribution, drawdown.
Spillover Engine

NVIDIA/OpenAI-compatible or local option.

2. Component Responsibilities

Component Responsibilities Does not do
Dashboard frontend Render import choices, risk cards, charts, tables, copilot panel, fallback states. Calculate authoritative risk metrics.
Backend API Coordinate import, validation, cache lookup, calculation requests, and response formatting. Store secrets in source code or expose raw access tokens.
Risk engine Compute deterministic financial risk metrics from normalized inputs. Generate trade recommendations.
Commodity engine Estimate commodity factor sensitivity and lagged spillover signals. Claim causality when only correlation/regression is measured.
News engine Fetch RSS metadata, tag risk categories, link headlines to sectors/holdings. Scrape full articles or act as the primary risk model.
Copilot Explain risk outputs, answer questions, generate a memo from structured context. Invent calculations or issue buy/sell instructions.

3. Primary Data Flow

User
  -> Dashboard import action
  -> Backend portfolio controller
  -> Zerodha / CSV / Sample loader
  -> Holding normalizer and sector mapper
  -> Price/factor/news cache lookup
  -> Risk engine + commodity engine + news engine
  -> Structured risk response
  -> Dashboard cards/charts
  -> Optional copilot question
  -> Risk MCP tools / LLM explanation
  -> Safe risk answer
Design principle The AI path is downstream from the analytics path. If the LLM provider is unavailable, the dashboard and deterministic risk memo should still work.

4. Zerodha Integration Flow

Prototype flow
  1. User clicks "Connect Zerodha" or chooses sample/CSV.
  2. Backend redirects to Zerodha login if live import is used.
  3. Callback receives request token and creates short-lived access session.
  4. Backend fetches holdings/positions and immediately normalizes them.
  5. Token is kept only for the active local session, then discarded.
Production flow
  1. User sees consent screen explaining exact data access.
  2. OAuth/session state is protected against CSRF and replay.
  3. Access tokens are encrypted and retained only as long as needed.
  4. Refresh/background jobs respect API rate limits and market data terms.
  5. Every account-link and data-access event is auditable.
External API Use in design
GET /portfolio/holdings Primary source for current equity/ETF holdings.
GET /portfolio/positions Detect intraday or overnight positions; deep modeling deferred in V1 if non-equity/F&O.
GET /instruments Map trading symbols to instrument tokens and metadata.
GET /quote, /quote/ohlc, /quote/ltp Latest prices where available and permitted.
GET /instruments/historical/:instrument_token/:interval Historical candles if paid/allowed path is available.
wss://ws.kite.trade Production/paid enhancement for live market updates.

5. MCP Architecture

The demo uses two MCP stories: Zerodha's Kite MCP for account/market connectivity in an AI environment, and a custom Risk MCP server for project-specific analytics.

Kite MCP

  • Demonstrates that broker data can be exposed to AI tools with controlled permissions.
  • Useful for live portfolio or market questions in an MCP-capable client.
  • Kept separate from the dashboard's reliability path.

Risk MCP

  • Exposes this project's risk functions as tools.
  • Calls deterministic analytics modules.
  • Returns structured JSON for LLM explanation and memo generation.
AI Client / Copilot
  -> Kite MCP: read Zerodha portfolio/market context
  -> Risk MCP: calculate VaR, ES, stress, spillover, news risk
  -> LLM: convert structured outputs into safe explanation

6. Deployment View

Local course demo
  • Frontend and backend run on localhost.
  • SQLite/files store cached prices, factors, news, and sample portfolios.
  • Secrets live in local .env.
  • Only presenter machine sees raw portfolio data.
  • Risk MCP runs as a local process.
  • LLM use can be disabled, local, or configured through a temporary API key.
Hosted product
  • Frontend hosted separately from backend API.
  • Backend deployed as a persistent service with background workers.
  • Postgres/Supabase or equivalent database with tenant separation.
  • Secrets managed through cloud secret manager.
  • Encrypted tokens, consent logs, audit trails, and deletion workflow.
  • Risk MCP secured behind authenticated tool access.

7. Security And Compliance View

Concern Prototype handling Production handling
User consent Presenter verbally explains what will be accessed. Consent screen with explicit data scopes and retention policy.
Token storage No persistent token storage; local active session only. Encrypted token storage, expiry, revocation, and short retention.
Market data terms Use local/cached/public data; avoid public redistribution. Licensed data strategy and exchange/broker term review.
Investment advice Disclaimers and copilot policy to avoid recommendations. Legal/SEBI review before monetized personalized usage.
LLM data exposure Send summarized risk context only, preferably with user-controlled key. Data minimization, provider contracts, redaction, audit logs.

8. Scalability And Cost View

Zero-cost prototype

  • Local machine only.
  • Sample/cached prices.
  • RSS headline metadata.
  • Optional free-tier LLM key.

Low-cost beta

  • Static frontend hosting.
  • Small backend server.
  • Postgres free/low tier.
  • Manual rate-limit controls.

Real product

  • Paid market data plan where needed.
  • Queue workers for refreshes.
  • Encrypted secret management.
  • Monitoring and compliance review.

9. Failure Handling Strategy

Failure User-facing behavior System behavior
Zerodha auth fails Show reconnect, CSV upload, and sample portfolio options. Do not retry indefinitely; log sanitized error.
Price data missing Show partial modeling warning and excluded symbols. Compute on available modeled holdings only.
RSS feed unavailable Show "news context unavailable" state. Do not block risk calculations.
LLM unavailable Show deterministic risk memo instead of chat answer. Skip provider call and keep dashboard functional.
MCP server unavailable Show copilot tool error and keep dashboard active. Restart local MCP process or disable AI tool path.
